# چالش کتابخانه‌ی من (۵۰۰ امتیاز)

توی این پوشه یک اپلیکیشن کوچک به اسم "کتابخانه‌ی من" پیدا می‌کنید. این اپلیکیشن به کاربر اجازه می‌ده کتاب‌هاش رو مدیریت کنه، کتاب اضافه کنه، ویرایش کنه، حذف کنه و توش جستجو کنه.

پروژه با **Node.js + Express** برای بک‌اند و **HTML + CSS + JavaScript خالص** (بدون هیچ فریمورکی) برای فرانت‌اند نوشته شده.

<br/>

## هدف چالش

این اپلیکیشن **حداقل ۱۰ باگ** داره. بعضیاشون به محض اجرا کاملاً واضحن، بعضیا فقط وقتی سناریوهای خاص رو تست کنید مشخص می‌شن، و بعضیا فقط با خوندن دقیق کد پیدا می‌شن.

## رفتار مورد انتظار

وقتی همه‌ی باگ‌ها fix شده باشن، اپلیکیشن باید بتونه این کارها رو درست انجام بده:

- لیست کتاب‌ها رو در صفحه نشون بده.
- دکمه‌ی "افزودن کتاب" یک مودال باز کنه که فرم داشته باشه. مودال در ابتدای صفحه نباید باز باشه.
- با ثبت فرم، کتاب جدید اضافه بشه و صفحه بدون رفرش لیست جدید رو نشون بده.
- کلیک روی ویرایش، مودال رو با اطلاعات همون کتاب باز کنه.
- کلیک روی حذف، کتاب رو حذف کنه و لیست بلافاصله بدون رفرش صفحه به‌روز بشه.
- جستجو با تایپ در باکس بالای صفحه انجام بشه و شامل هر کاراکتری (از جمله پرانتز و براکت) بدون خطا کار کنه.
- سرور با درخواست‌های نامعتبر (id ناموجود، بدنه‌ی خالی، ...) کرش نکنه و پیام مناسب برگردونه.

وظیفه‌ی شما اینه:

1. باگ‌ها رو پیدا کنید.
2. برای هر باگ یک fix بنویسید.
3. برای هر باگ در فایل `BUGS.md` یک بخش کوتاه (۳-۵ خط) بنویسید که شامل این موارد باشه:
   - **کجای کد**: مسیر فایل و شماره‌ی خط تقریبی
   - **مشکل چی بود**: به زبان ساده توضیح بدید
   - **چرا این‌طور شد**: ریشه‌ی مشکل کجاست (نه فقط علامت ظاهری)
   - **چطور درست کردید**: خلاصه‌ی راه‌حل

### قالب پیشنهادی برای BUGS.md

```md
## باگ ۱ — [یک اسم کوتاه برای باگ]

- کجا: [مسیر فایل] خط [شماره]
- مشکل: [یک جمله درباره‌ی رفتار نادرست]
- ریشه: [چرا این رفتار اتفاق افتاده — دلیل فنی، نه فقط علامت]
- fix: [خلاصه‌ی چند کلمه‌ای تغییری که دادید]
```

<br/>

## راه‌اندازی

```bash
cd bug-hunt
npm install
npm start
```

بعد توی مرورگر `http://localhost:3000` رو باز کنید.

قبل از شروع باگ‌یابی، پیشنهاد می‌کنیم فایل [`ARCHITECTURE.md`](./ARCHITECTURE.md) رو یک بار بخونید تا با ساختار پروژه و مسئولیت هر فایل آشنا بشید.

<br/>

## دسته‌بندی باگ‌ها

باگ‌ها توی این پروژه از این جنس‌ها هستن (بعضی از این دسته‌ها ممکنه چند باگ داشته باشن، بعضی هیچ‌کدوم):

- منطق جاوااسکریپت و DOM
- CSS و ظاهر
- ارتباط بین فرانت و بک (fetch, form)
- منطق سرور و روت‌ها
- مدیریت خطا و امنیت اولیه

<br/>

## چند تا از باگ‌های واضح‌تر

برای اینکه شروع کارتون راحت‌تر باشه، ۶ تا از باگ‌های ساده‌تر و قابل مشاهده‌تر رو اینجا لیست کردیم. این‌ها فقط علامت ظاهری باگ هستن — ریشه‌ی مشکل و راه‌حل رو خودتون باید پیدا کنید:

1. **مودال همیشه باز است.** به محض باز کردن صفحه، فرم افزودن/ویرایش کل صفحه رو پوشونده و اجازه‌ی هیچ کاری نمی‌ده. باید مخفی باشه و فقط با دکمه‌ی "افزودن" یا "ویرایش" باز بشه.

2. **لیست کتاب‌ها نشون داده نمی‌شه.** با اینکه سرور کتاب‌ها رو به‌درستی برمی‌گردونه (می‌تونید توی تب Network این رو ببینید)، صفحه‌ی اصلی خالیه.

3. **ثبت فرم باعث رفرش شدن صفحه می‌شه.** بعد از پر کردن فرم افزودن کتاب و زدن دکمه‌ی ذخیره، صفحه رفرش می‌شه و ممکنه اطلاعات از دست بره یا رفتار غیرمنتظره‌ای ببینید.

4. **بعد از حذف کتاب، کتاب هنوز روی صفحه هست.** دکمه‌ی حذف رو می‌زنید، به بک‌اند درخواست حذف می‌ره، ولی خود کتاب توی UI نمی‌ره تا وقتی صفحه رفرش بشه.

5. **جستجو با کاراکترهای خاص خطا می‌ده.** توی باکس جستجو کافیه یک `(` یا `[` تایپ کنید تا سرور به جای لیست کتاب‌ها یک صفحه‌ی خطا (stack trace) برگردونه.

6. **ویرایش یک کتاب که وجود نداره (یا آی‌دی اشتباه) خطای ۵۰۰ می‌ده.** اگه از ابزارهایی مثل curl یا Postman یک PUT به یک آی‌دی ناموجود بزنید، سرور به جای گفتن "پیدا نشد" یک stack trace کامل به کاربر برمی‌گردونه.

<br/>

**نکته:** این ۶ تا همه‌ی باگ‌های پروژه نیستن. حداقل ۴ باگ دیگه هم توش هست که توی این لیست نیومده — بعضیاشون فقط با خوندن دقیق کد یا تست کردن سناریوهای خاص پیدا می‌شن. هر باگ اضافی که پیدا کنید و درست fix کنید امتیاز داره.

<br/>

## امتیازدهی

- **۳۰ امتیاز** برای هر باگ که هم پیدا کردید و هم درست fix کردید (با توضیح درست ریشه). جمع = ۳۰۰ امتیاز.
- **۱۵۰ امتیاز** برای کیفیت و دقت توضیحاتتون توی `BUGS.md`. اینجا خیلی مهمه که "چرا این باگ اتفاق افتاده" رو بلد باشید، نه فقط "چطور مخفیش کردم".
- **۵۰ امتیاز** اگه باگ‌های اضافه‌تری (که ما تو لیستمون نبودن) هم پیدا کنید و درست کنید.

<br/>

## نکات

- تعداد باگ‌ها **حداقل ۱۰** تاست. اگه بیشتر پیدا کردید، امتیاز اضافه می‌گیرید.
- بعضی باگ‌ها با چشم دیده می‌شن، بعضیا فقط با کلیک روی چند تا دکمه و تست کردن سناریوهای مختلف مشخص می‌شن.
- خواندنی بودن fix هم مهمه. راه‌حل‌های تمیز‌تر امتیاز بیشتر دارن.
- لطفاً کد رو کاملاً بازنویسی نکنید. سعی کنید کمترین تغییر ممکن رو بدید که باگ رو حل کنه.
- برای افزایش یادگیری و دقیق‌تر بودن ارزیابی لطفاً از هوش مصنوعی برای پیدا کردن یا حل باگ‌ها استفاده نکنید. توی مصاحبه ازتون می‌خوایم یک باگ رو زنده روی سیستم خودتون توضیح بدید.

<br/>

## ارسال

پروژه رو با تغییرات خودتون (و فایل `BUGS.md`) داخل پوشه‌ی `bug-hunt/` قرار بدید. اگه git history داشتید (که پیشنهاد می‌کنیم داشته باشید — هر باگ رو یک commit جدا) اون هم داخلش باشه.
