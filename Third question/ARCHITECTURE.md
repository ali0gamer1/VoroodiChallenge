# ساختار پروژه

این سند ساختار کد این پروژه رو توضیح می‌ده تا قبل از شروع باگ‌یابی یک نقشه‌ی کلی از چیدمان فایل‌ها و مسیر جریان داده داشته باشید.

<br/>

## نگاه کلی

پروژه یک اپلیکیشن ساده‌ی مدیریت کتاب‌ست که از دو قسمت تشکیل شده:

- **بک‌اند** با Node.js و Express. داده رو در یک فایل JSON نگه می‌داره.
- **فرانت‌اند** با HTML، CSS و JavaScript خالص (بدون فریمورک، بدون build step).

هر دو قسمت توسط همون سرور Express سرو می‌شن. یعنی وقتی مرورگر `http://localhost:3000` رو باز کنه، Express فایل‌های استاتیک داخل `public/` رو برمی‌گردونه.

<br/>

## درخت فایل‌ها

```
bug-hunt/
├── server.js              نقطه‌ی ورود سرور. Express رو راه می‌ندازه و روت‌ها/میدلورها رو mount می‌کنه.
├── db.js                  توابع خواندن و نوشتن روی db.json.
├── db.json                داده‌ی پایدار. لیست کتاب‌ها + شناسه‌ی بعدی.
├── package.json           وابستگی‌ها و اسکریپت‌های npm.
│
├── middleware/
│   └── logger.js          یک میدلور ساده که هر درخواست رو در console چاپ می‌کنه.
│
├── routes/
│   └── books.js           چهار روت مربوط به کتاب: GET, POST, PUT, DELETE.
│
└── public/                فایل‌های استاتیک که مرورگر مستقیماً بارگذاری می‌کنه.
    ├── index.html         ساختار HTML صفحه. مدال فرم و لیست کتاب‌ها.
    ├── style.css          استایل‌های سراسری.
    └── js/
        ├── api.js         توابع wrapper روی fetch برای صدا زدن APIهای بک‌اند.
        ├── modal.js       باز و بسته کردن مودال و نگهداری وضعیت "در حال ویرایش".
        ├── render.js      رندر کردن لیست کتاب‌ها و اتصال هندلر به دکمه‌های کارت‌ها.
        └── app.js         سیم‌کشی نهایی: هندلرهای فرم و جستجو، و صدا زدن اولیه‌ی loadBooks.
```

<br/>

## بک‌اند

### مسئولیت هر فایل

| فایل | مسئولیت |
| --- | --- |
| `server.js` | ساخت اپ Express، تنظیم میدلورهای عمومی، mount کردن روت‌ها، شروع listen. |
| `db.js` | ارائه‌ی `readDB()` و `writeDB(data)`. تنها لایه‌ای که با فایل‌سیستم کار می‌کنه. |
| `middleware/logger.js` | یک تابع میدلور که متد، مسیر و تایم‌استمپ درخواست رو لاگ می‌کنه. |
| `routes/books.js` | یک Express Router که ۴ روت زیر رو تعریف می‌کنه: `GET /`, `POST /`, `PUT /:id`, `DELETE /:id`. |

### روت‌های API

روت‌ها زیر `/api/books` mount شدن، پس آدرس نهایی هر روت اینه:

| متد | مسیر | کاربرد |
| --- | --- | --- |
| `GET` | `/api/books` | لیست همه‌ی کتاب‌ها. با query param `?q=...` جستجو می‌کنه. |
| `POST` | `/api/books` | اضافه کردن کتاب جدید. بدنه: `{ title, author }`. |
| `PUT` | `/api/books/:id` | ویرایش کتاب. بدنه: `{ title, author, read }`. |
| `DELETE` | `/api/books/:id` | حذف کتاب. |

### مدل داده

هر کتاب این شکل رو داره:

```json
{
  "id": 1,
  "title": "بوف کور",
  "author": "صادق هدایت",
  "read": true
}
```

فایل `db.json` هم یک فیلد `nextId` نگه می‌داره که هنگام ساختن کتاب جدید به عنوان شناسه استفاده می‌شه و بعدش یکی زیاد می‌شه.

## فرانت‌اند

### ترتیب بارگذاری اسکریپت‌ها

`index.html` چهار اسکریپت رو به این ترتیب load می‌کنه:

```html
<script src="js/api.js"></script>
<script src="js/modal.js"></script>
<script src="js/render.js"></script>
<script src="js/app.js"></script>
```

اسکریپت‌ها classic هستن (نه module). یعنی متغیرهای top-level بین همه‌ی اسکریپت‌ها قابل دسترسیه. برای همین `openModal` که در `modal.js` تعریف شده، در `render.js` هم قابل استفاده‌ست، بدون هیچ import ای.

ترتیب بارگذاری اهمیت داره چون هر فایل ممکنه به توابع فایل‌های قبلی نیاز داشته باشه:

- `render.js` از `openModal` و `deleteBook` استفاده می‌کنه.
- `app.js` از `render`, `getBooks`, `createBook`, `updateBook`, `openModal`, `closeModal`, `form`, `editingId` استفاده می‌کنه.

### مسئولیت هر فایل

| فایل | مسئولیت |
| --- | --- |
| `index.html` | ساختار DOM شامل هدر، لیست کتاب‌ها، پیام "لیست خالی" و مودال فرم. |
| `style.css` | تمام استایل‌های صفحه. یک فایل تخت بدون تفکیک ماژول‌ها. |
| `js/api.js` | چهار تابع `getBooks`, `createBook`, `updateBook`, `deleteBook`. تنها لایه‌ای که مستقیم با fetch کار می‌کنه. |
| `js/modal.js` | نگه‌داری ref به مودال و فرم، مدیریت متغیر `editingId`، تابع `openModal(book)` و `closeModal()`. |
| `js/render.js` | تابع `render(books)` که لیست رو در صفحه می‌کشه و روی هر کارت هندلر ویرایش و حذف می‌ذاره. |
| `js/app.js` | صدا زدن `loadBooks` اولیه، سیم‌کشی هندلر submit فرم، هندلر تایپ در باکس جستجو و کلیک روی دکمه‌های "افزودن" و "انصراف". |

<br/>

## جریان درخواست (Request lifecycle)

برای اینکه ذهنیت روشن‌تری بگیرید، یک سناریوی کامل رو دنبال می‌کنیم:

**سناریو: کاربر یک کتاب جدید اضافه می‌کنه.**

1. کاربر روی "افزودن کتاب" کلیک می‌کنه → `app.js` تابع `openModal(null)` رو از `modal.js` صدا می‌زنه → مودال باز می‌شه با فرم خالی.
2. کاربر فرم رو پر می‌کنه و "ذخیره" رو می‌زنه.
3. `app.js` هندلر submit فرم رو اجرا می‌کنه. یک payload می‌سازه و `createBook(payload)` رو از `api.js` صدا می‌زنه.
4. `api.js` یک `POST /api/books` می‌فرسته.
5. سرور روت `POST /` رو در `routes/books.js` اجرا می‌کنه. `readDB` رو صدا می‌زنه، کتاب جدید رو به آرایه اضافه می‌کنه، `writeDB` رو صدا می‌زنه، جواب می‌ده.
6. `app.js` بعد از موفقیت `closeModal()` رو می‌زنه و `loadBooks(searchInput.value)` رو صدا می‌کنه.
7. `loadBooks` تابع `getBooks` از `api.js` رو می‌زنه، جواب رو به `render` از `render.js` می‌ده.
8. `render` DOM رو خالی می‌کنه و کارت‌های جدید می‌کشه.

<br/>

## چند نکته‌ی فنی

- **بدون build step**: مرورگر مستقیم فایل‌های `public/` رو اجرا می‌کنه. تغییری در JS یا CSS دادید، فقط صفحه رو رفرش کنید.
- **بدون فریمورک**: تمام کار DOM با API استاندارد مرورگر (`document.getElementById`, `addEventListener`, `createElement`) انجام می‌شه.
- **پایگاه داده فایل‌ست**: `db.json` بعد از هر عملیات نوشته می‌شه. اگه دیتا خراب شد، همیشه می‌تونید فایل رو به حالت اولیه برگردونید.
- **Express رفتار خطاها**: سرور Express بعضی از خطاهای همزمان درون هندلرها رو خودش می‌گیره و یک پاسخ `500` با stack trace می‌فرسته. یعنی همه‌ی خطاها لزوماً باعث "کرش کامل" سرور نمی‌شن.
